# Отчет по результатам автоматизации тестирования приложения FMHandroid

## 1. Сравнение ручного и автоматизированного тестирования

В рамках проекта было проведено сравнение эффективности ручного прогона тест-кейсов и выполнения разработанных автоматизированных UI-тестов.


| Критерий сравнения | Ручное тестирование (по тест-кейнам) | Автоматизированное тестирование (UI-тесты) |
| :--- | :--- | :--- |
| **Затраченное время** | от 1 часа до 1.5 часов (60–90 минут) | 17 минут 33 секунды |
| **Участие тестировщика** | Требует 100% личного времени и внимания | Выполняется автоматически в фоновом режиме |
| **Риск пропустить дефект** | Высокий (влияет человеческий фактор и усталость) | Исключен (робот строго следует шагам кода) |
| **Генерация отчетов** | Составляется вручную после всех проверок | Allure-отчет формируется автоматически |

**Вывод:** Автоматизация позволила сократить время тестирования приложения более чем в 4 раза, полностью освободив инженера от рутинных однотипных проверок.

## 2. Найденные проблемы и дефекты приложения

В ходе ручного анализа и стабилизации автотестов в приложении FMHandroid был локализован и задокументирован комплекс дефектов логики и интерфейса (UX/UI):

1. **Критический функциональный баг навигации (Тест №23):** Обнаружен сбой при попытке перейти в раздел "О приложении" (About) непосредственно с экрана "Новости" (News). Навигационный сценарий блокируется, что подтверждается единственным упавшим тест-кейсом в Allure-отчете.
2. **UX-дефект экрана авторизации (Класс LoginTest):** При вводе любых невалидных учетных данных (неверный регистр, спецсимволы, несуществующий аккаунт) система выводит неинформативное Toast *"Something went wrong. Try again later."*. Данный текст имитирует поломку сервера, дезориентируя пользователя вместо вывода понятного уведомления о неверно введенных учетных данных (например, *"Неверный логин или пароль"*).
3. **UX-дефект редактора новостей (Тест №56):** При попытке ручного ввода категории вместо выбора из выпадающего списка, приложение выдает системную ошибку *"Saving failed. Try again later."*. Это свидетельствует о падении операции на сервере вместо корректной клиентской валидации поля с понятным пользователю предупреждением (например, *"Выберите категорию из списка"*).
4. **UX-дефект формы создания новостей:** Аналогичный дефект валидации дублируется на изолированном экране создания новой новости. При попытке прописать категорию вручную клавиатурой система отправляет некорректный запрос и падает с ошибкой сервера *"Saving failed..."*, не блокируя некорректный ввод на клиенте.
5. **Дефект логики фильтрации в Новостях (Тест №42 в NewsFilterTest):** При вводе инвертированного периода (дата начала позже даты окончания) приложение пропускает некорректный запрос и отображает пустую страницу вместо вывода предусмотренной логикой ошибки *"Wrong period"*.
6. **Дефект логики фильтрации в Панели управления (Тест №84 в ControlPanelFilterTest):** Дублирование предыдущей ошибки на изолированном экране фильтрации Control Panel. Система также обрабатывает логически неверный диапазон дат и открывает пустой экран без предупреждения пользователя.
7. **Дефект сохранения состояния фильтров при навигации:** Приложение полностью сбрасывает настроенные пользователем фильтры новостей, если он переключается на другой экран (например, в раздел "Main"). При возвращении в новости принудительно загружается весь список, что портит пользовательский опыт (UX).
8. **Отсутствие ограничений на длину ввода (maxLength):** В форме создания новости поля заголовка и описания позволяют вводить неограниченное количество символов. На этапе отображения интерфейс обрабатывает это корректно (заголовок усекается троеточием, описание расширяется), однако отсутствие лимитов при вводе создает уязвимость для переполнения базы данных.

## 3. Трудности, возникшие при автоматизации

В процессе разработки и стабилизации UI-тестов были успешно решены следующие технические проблемы:

1. **Зависимость тестов от состояния сессии (авторизация):** Изначально тесты падали при запуске цепочкой, если приложение уже находилось в авторизованном состоянии. Проблема решена внедрением жесткой изоляции в методах `@Before` через проверку статуса `if-else` и принудительный сброс состояния через `logout()`.
2. **Асинхронность Espresso и преждевременное считывание данных:** Тесты на отмену даты и времени (№63, №64, №66, №68) падали, так как робот пытался прочитать текст до фактической загрузки формы редактора. Ошибка устранена за счет изменения порядка шагов и добавления обязательного ожидания `checkEditFormLoaded()`.
3. **Проверка граничных значений текстовых полей:** Было протестировано поведение интерфейса при вводе избыточного количества символов в поля заголовка и описания новости. Корректность верстки UI при этом не нарушается: длинный заголовок в общем списке аккуратно сокращается с использованием троеточия (ellipsize), а поле описания корректно адаптируется по высоте и отображает весь текст при развертывании карточки.

## 4. Централизация данных и Использованный стек

* **Управление конфигурацией:** Все повторяющиеся учетные данные (валидный логин и пароль) были успешно вынесены в статические методы класса `DataHelper`. Это позволяет изменять параметры авторизации централизованно в одном файле.
* **Технологический стек:** Java 8 / JUnit 4 / Espresso / UI Automator (для работы с системными компонентами Android) / Allure Framework / Page Object Model (POM).
